### 사례 - 설문 상세 정보 조회

![image](https://github.com/user-attachments/assets/dba42d2c-9a39-4810-afae-dc54abd8cd9b)

설문의 상세 정보를 조회하기 위해서는 **DB에서 설문 정보와 참여자 수(추첨 보드)를 조회**해야한다.

설문 정보와 참여자 수를 조회하는 순서는 상관 없다.

그러므로 설문 정보 조회와 참여자 수 조회를 비동기적으로 호출하는 동시성 처리를 통해 성능을 향상시킬 수 있다.

### 동시성 처리를 위한 방법 - Coroutine(kotlin), Completable Future

설문 상세 정보를 동기적으로, 비동기적으로(Coroutine, Completable Future) 호출하여 성능을 비교해보자.

- **동기**
    
    ```kotlin
    fun getSurveyInfo1(surveyId: UUID): SurveyInfoResponse {
        val startTime = System.currentTimeMillis()
    
        val survey = surveyAdapter.getSurvey(surveyId)
        val surveyFetchTime = System.currentTimeMillis()
        log.info("S: ${surveyFetchTime - startTime}")
    
        if (survey.status == SurveyStatus.NOT_STARTED) throw InvalidSurveyAccessException()
    
        val selectedTicketCount =
            if (survey.rewardInfo.isImmediateDraw) {
                val drawingStartTime = System.currentTimeMillis()
    
                val ticketCount = drawingBoardAdapter.getBySurveyId(surveyId).selectedTicketCount
    
                val drawingFetchTime = System.currentTimeMillis()
                log.info("B: ${drawingFetchTime - drawingStartTime}")
    
                ticketCount
            } else {
                null
            }
    
        val totalTime = System.currentTimeMillis() - startTime
        log.info("T: $totalTime\n")
    
        return SurveyInfoResponse.of(survey, selectedTicketCount)
    }
    ```
    
- **비동기(Couroutine)**
    
    ```kotlin
    fun getSurveyInfo2(surveyId: UUID): SurveyInfoResponse =
        runBlocking {
            val startTime = System.currentTimeMillis()
    
            // 설문 조회와 추첨 보드 조회를 비동기로 동시에 처리
            val surveyDeferred =
                async(Dispatchers.IO) {
                    val surveyStartTime = System.currentTimeMillis()
                    val survey =
                        withContext(Dispatchers.IO) {
                            surveyAdapter.getSurvey(surveyId) // 동기 작업을 IO 디스패처에서 실행
                        }
                    val surveyEndTime = System.currentTimeMillis()
                    log.info("S: ${surveyEndTime - surveyStartTime}")
                    survey
                }
    
            val drawingBoardDeferred =
                async(Dispatchers.IO) {
                    val drawingStartTime = System.currentTimeMillis()
                    val drawingBoard =
                        withContext(Dispatchers.IO) {
                            drawingBoardAdapter.getBySurveyId(surveyId) // 동기 작업을 IO 디스패처에서 실행
                        }
                    val drawingEndTime = System.currentTimeMillis()
                    log.info("B: ${drawingEndTime - drawingStartTime}")
                    drawingBoard
                }
    
            // 설문 데이터와 추첨 보드 데이터를 병렬로 대기
            val survey = surveyDeferred.await()
            val drawingBoard = drawingBoardDeferred.await()
    
            // 설문 상태 체크
            if (survey.status == SurveyStatus.NOT_STARTED) throw InvalidSurveyAccessException()
    
            // 추첨 보드 조회 결과 처리
            val selectedTicketCount =
                if (survey.rewardInfo.isImmediateDraw) {
                    drawingBoard.selectedTicketCount
                } else {
                    null
                }
    
            // 총 처리 시간 로깅
            val totalTime = System.currentTimeMillis() - startTime
            log.info("T: $totalTime\n")
    
            SurveyInfoResponse.of(survey, selectedTicketCount)
        }
    ```
    
- **비동기(Completable Future)**
    - 참고 - 야놀자 개발자 이력서
        
        ![image](https://github.com/user-attachments/assets/deb5f44e-577a-430e-a070-303801177d23)
        
        이력서 가장 상단에 있음, 실무에서도 많이 사용된다고 들음
        
    
    ```kotlin
    fun getSurveyInfo3(surveyId: UUID): SurveyInfoResponse {
        val startTime = System.currentTimeMillis()
    
        // 설문 조회 CompletableFuture
        val surveyFuture =
            CompletableFuture.supplyAsync {
                val surveyStartTime = System.currentTimeMillis()
                val survey = surveyAdapter.getSurvey(surveyId) // 동기 작업을 비동기처럼 처리
                val surveyEndTime = System.currentTimeMillis()
                log.info("S: ${surveyEndTime - surveyStartTime}")
                survey
            }
    
        // 추첨 보드 조회 CompletableFuture
        val drawingBoardFuture =
            CompletableFuture.supplyAsync {
                val drawingStartTime = System.currentTimeMillis()
                val drawingBoard = drawingBoardAdapter.getBySurveyId(surveyId) // 동기 작업을 비동기처럼 처리
                val drawingEndTime = System.currentTimeMillis()
                log.info("B: ${drawingEndTime - drawingStartTime}")
                drawingBoard
            }
    
        // 두 CompletableFuture를 병렬로 대기
        val survey = surveyFuture.join()
        val drawingBoard = drawingBoardFuture.join()
    
        // 설문 상태 체크
        if (survey.status == SurveyStatus.NOT_STARTED) throw InvalidSurveyAccessException()
    
        // 추첨 보드 조회 결과 처리
        val selectedTicketCount =
            if (survey.rewardInfo.isImmediateDraw) {
                drawingBoard.selectedTicketCount
            } else {
                null
            }
    
        // 총 처리 시간 로깅
        val totalTime = System.currentTimeMillis() - startTime
        log.info("T: $totalTime\n")
    
        return SurveyInfoResponse.of(survey, selectedTicketCount)
    }
    ```
    

### 테스트 환경

SpringBoot + Kotlin + MongoDB(설문 & 추첨 보드 16000 row)

세 가지 방식으로 구현한 설문 상세 정보 API를 100번 호출하여 응답 시간 측정

### 결과 - 약 20%의 성능 개선이 이루어짐

|  | 설문 조회 | 추첨 보드 조회 | total |
| --- | --- | --- | --- |
| 동기 | 21.91 ms | 151.91 ms | 175.44 ms |
| 비동기(Coroutine) | 19.75 ms | 138.45 ms | 140.74 ms |
| 비동기(Completable Future) | 24.96 | 139.67 ms | 141.16 ms |

현재 상황에서는 큰 효과를 보진 못했지만, 여러번의 DB 호출이 순서 상관없이 이루어져도 되고, DB의 응답 시간이 길다면 더 큰 효과가 있을 것이다.

참고로, 추첨 보드 조회가 훨씬 오래걸리는 이유는 index 조회가 아니기 때문이다.

### 부록 1 - Coroutine은 내부적으로 어떻게 동작할까? (feat. Thread, Context Switch)

아까 테스트한 Coroutine 코드를 기준으로 보면

```kotlin
// DB 설문 조회 작업
val surveyDeferred =
    async(Dispatchers.IO) {
		    withContext(Dispatchers.IO) {
				    surveyAdapter.getSurvey(surveyId)
        }
    }

// DB 추첨 보드 조회 작업
val drawingBoardDeferred =
    async(Dispatchers.IO) {
		    withContext(Dispatchers.IO) {
				    drawingBoardAdapter.getBySurveyId(surveyId)
        }
    }
```

두 DB 조회 작업은 각각 Dispatchers.IO에서 IO 바운드 작업을 처리하기 위한 쓰레드를 할당받는다.

CPU는 두 쓰레드 중 하나의 IO 작업(DB 조회)을 진행하는 시스템 콜을 발생시키고, 이 작업이 완료될 때 까지 남는 시간 동안 다른 쓰레드로 컨텍스트 스위칭을 하고, 다른 쓰레드의 작업을 진행한다.

### 부록 2 - Completable Future는 내부적으로 어떻게 동작할까?

아까 테스트한 Completable Future 코드를 기준으로 보면

```kotlin
// 설문 조회 CompletableFuture
val surveyFuture =
    CompletableFuture.supplyAsync { surveyAdapter.getSurvey(surveyId) }

// 추첨 보드 조회 CompletableFuture
val drawingBoardFuture =
        CompletableFuture.supplyAsync { drawingBoardAdapter.getBySurveyId(surveyId) }
```

Coroutine과 비슷하게, 두 DB 조회 작업은 각각 ForkJoinPool에서 쓰레드를 할당받는다.

또 Coroutine과 마찬가지로, CPU는 두 쓰레드 중 하나의 IO 작업(DB 조회)을 진행하는 시스템 콜을 발생시키고, 이 작업이 완료될 때 까지 남는 시간 동안 다른 쓰레드로 컨텍스트 스위칭을 하고, 다른 쓰레드의 작업을 진행한다.

### 부록 3 - Coroutine과 Completable Future의 차이점

Couroutine은 중단과 재개 기능을 통해 단일 쓰레드에서도 여러 코루틴을 효율적으로 처리할 수 있고, 같은 쓰레드에서 작업이 재개될 확률이 높다. 이로 인해 Completable Future보다 자원을 더 효율적으로 사용할 수 있다.

또한, Coroutine은 경량 쓰레드이므로 Completable Future 보다 동시성 처리의 효율이 더 좋다.
